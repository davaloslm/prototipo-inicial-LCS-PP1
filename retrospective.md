# Retrospectivas del proyecto LogiTrack

## Sprint 1 
###  Empezar a hacer:
- Usar los objetivos SMART, el alcance/no alcance y la selección del MVP como filtro real para priorizar el backlog inicial.
- Vincular cada historia de usuario con un criterio de aceptación basado en la Ley 25.326.
- Revisar el registro de riesgos en cada planning.

### Hacer más:
- Reuniones de alineación para asegurar que el equipo encaré el proyecto con el mismo enfoque.
- Identificación y seguimiento temprano del registro de riesgos para que no quede sólo como entregable documental.

### Seguir haciendo:
- Reuniones y comunicaciones diarias para hablar sobre el proyecto.
- Construir una base de trabajo ordenada con tablero, plan de comunicación, working agreement y definiciones de ready/done.

### Dejar de hacer:
- Tratar backlog, riesgos, comunicación y alcance como documentos separados sin relación entre sí.
- Postergar el análisis legal para semanas posteriores.

### Hacer menos:
- Tiempo invertido en redactar documentos sin convertirlos en decisiones operativas del sprint.

## Sprint 2 
### Empezar a hacer:
- Hacer un seguimiento entre el prototipo, el modelo de dominio y NFRs antes de cada entrega

### Hacer más:
- Revisión funcional del prototipo para asegurar las funcionalidades de envío.

### Seguir haciendo:
- Consolidar el repositorio, teniendo el prototipo, la documentación y el trabajo colaborativo.
- Avanzar con un prototipo navegable visible que permita demostrar el MVP temprano.
- Documentar decisiones técnicas y legales con ADRs

### Dejar de hacer:
- Diseñar pantallas sin revisar si el flujo respeta el dominio y los estados definidos del envío.
- Dejar los NFRs como lista teórica sin relacionarlos con el comportamiento esperado del sistema.
- Separar el plan de pruebas del desarrollo real de las funcionalidades.

### Hacer menos:
- Cambios visuales que no agregan valor si todavía faltan cerrar flujos centrales del prototipo.

##  Sprint 3
### Empezar a hacer:
- Seguir actualizando la matriz de trazabilidad
Hacer más:
- Redacción progresiva del informe, en lugar de concentrarla al cierre del sprint.
- Utilización de convenciones de commits.

### Seguir haciendo:
- Construir el modelo de machine learning para predecir prioridad de envíos.
- Mantener el repositorio ordenado con ramas entendibles, README explicativo del proyecto y commits que reflejen el tipo y el avance.
- Documentar el avance técnico y funcional en el informe y en la matriz de trazabilidad.

### Dejar de hacer:
- Integrar componentes tarde sin haber definido antes formato de datos, rutas y responsabilidades.
- Dejar CI como tarea secundaria.
- Redactar README e informe sólo al final, cuando ya cuesta reconstruir decisiones y evidencia.
### Hacer menos:
- Merges y push grandes con demasiados cambios.
